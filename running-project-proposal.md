## Note on the project shape — this book is unusual

The other "FIELD and AI" books in the workshop (compfin, corpfin, branding-and-ai) are domain-engineering books with deliverables like *a memo, a model, a campaign*. The intelligence book is a natural-history/philosophy book; the reader is *thoughtful general reader* (per `book.md`), not "engineering student building a financial model." The right project for this book has to feel like an *investigation*, not an assignment. The Extension Notes leitmotif (each chapter ends with a tool that extends the chapter's capacity, culminating in Ch 18) is the structural thread the project should ride.

The four options below are calibrated to this. Each one has a real artifact at the end the reader would actually want to keep. None of them require the reader to write code (though one of them lets them).

---

## Project Option 1: *Profile One Mind* — A Cognitive Shape

**What it is:** Pick one agent at the very start of the book — your dog, an octopus, a four-year-old you know, GPT-4, yourself — and across all 19 chapters build a comprehensive cognitive profile of that single agent, one capacity at a time.

**Final deliverable:** A 19-axis cognitive profile of one chosen agent, written in the book's voice — empirical where evidence permits, honest where it doesn't, with each chapter contributing a section on "what the agent does on this axis, what the diagnostic test would be, and what the evidence currently shows." By Ch 17 the reader has produced exactly the kind of "shape, not height" profile the AI chapter asks the reader to think with. By Ch 18 they have a personal Extension Note: which capacities of their chosen agent's profile have been extended by tools, and what the extensions changed.

**Why it fits this book:** The book's thesis from page one is that intelligence is a shape not a height, and that the question is which capacity in which agent under which test. A 19-axis profile of one agent *is* the book's argument applied to one case. It rewards close reading of the text — the diagnostic tests are in the chapters — and produces a deliverable that is uniquely the reader's.

**Adaptability:** A reader who picks their dog will spend Ch 11 on Horowitz and the guilty-look distinction; a reader who picks an octopus will spend Ch 12 on Finn's coconut-carrying and Ch 6 on distributed visual processing; a reader who picks GPT-4 will spend Ch 17 on the Ullman perturbations applied to whatever model they're running. Same project, very different deliverable, no template to dilute.

**Tool path:** Claude Project. Persistent context across sessions is the whole point — the reader returns to the same project across weeks and the system remembers their chosen agent, prior sections, and accumulated evidence. Ch 17 specifically wants the reader to use Claude (or another LLM) as both subject and instrument.

## Project Option 2: *The Sieve Toolkit* — Operational Tests for Intelligence

**What it is:** Treat each chapter as a method, not a description. From every chapter, extract the operational diagnostic — Sally-Anne, mirror mark test, Ullman perturbation, devaluation paradigm, etc. — and convert it into a copy-pastable testing protocol the reader could run on any new system. By the end of the book, the reader has a working toolkit: a portable set of tests, with each test naming what passes, what fails, and what the diagnostic distinguishes.

**Final deliverable:** A reference document — call it *Sieves* — with one page per chapter. Each page contains: the capacity, the operational definition, the diagnostic test in step-by-step form, the falsifiers (what would count as the system failing for the wrong reasons), and a worked example applying the test to one current AI system.

**Why it fits this book:** The book's central methodological move is converting vague concepts ("does the dog feel guilt?") into operational tests ("does the guilty-look behavior depend on actual misbehavior or on owner scolding?"). Ch 1 names this explicitly — definitions are sieves. The Sieve Toolkit is the book's methodology made portable. It also produces exactly the kind of artifact a working researcher, journalist, or product reviewer would actually use.

**Adaptability:** A psychology grad student will use the toolkit on lab animals; a product reviewer will use it on the latest model release; a teacher will use it as the basis for student exercises. The protocol is the same; the deployment varies.

**Tool path:** Claude chat for the per-chapter extractions, with optional Cowork to assemble the protocols into a single working document at the end. No code required, but Claude Code becomes useful at Ch 17 if the reader wants to script Ullman-style perturbation runs against an LLM.

## Project Option 3: *Extension Atlas* — A Personal Catalog of Cognitive Tools

**What it is:** Mirror the book's leitmotif. Each chapter ends with an Extension Note about a tool that extends the chapter's cognitive capacity beyond the biological limit. The reader builds the same atlas for *their own life* — chapter by chapter, identify a tool you actually use that extends the chapter's capacity, analyze how the extension works, and decide whether it strengthens, distorts, or replaces the underlying capacity. By Ch 18 the reader has documented their own extended mind.

**Final deliverable:** A personal *Extension Atlas* — 19 short essays, one per chapter, each cataloging one tool the reader uses, what cognitive capacity it extends, what it does well, what it costs, and what would be lost if the tool went away tomorrow. The Ch 18 capstone is a single integrated map of the reader's own cognitive perimeter — what is biological, what is extended, and where the seams are.

**Why it fits this book:** The Extension Notes are not decoration — book.md names them as the structural device that culminates in Ch 18. A project that makes the reader build their *own* Extension Notes is the most direct possible engagement with the book's spine. It also answers a question the book asks but does not finish: "what is the catalog like for *you*?"

**Adaptability:** A field biologist's atlas will look nothing like a software engineer's, which will look nothing like a parent's. The skeleton is shared — capacity, tool, mechanism, cost — and the entries are individual.

**Tool path:** Claude chat is sufficient. A Claude Project becomes useful if the reader wants to refer back across chapters to compare entries. No code required.

---

## Project Option 4: *Skeptic's Notebook on Frontier AI*

**What it is:** Use the book as a diagnostic instrument for current AI. Each chapter, take the chapter's capacity and run a diagnostic on a current frontier system. By the end, the reader has a chapter-by-chapter, evidence-based field report on what current LLMs and frontier AI systems can and cannot do, on the capacities the book defines and tests.

**Final deliverable:** A *Skeptic's Notebook* — 19 dated entries, each pairing a chapter's capacity with a diagnostic the reader runs against Claude/GPT/Gemini/whichever, with the system's response, the reader's evaluation against the chapter's criterion, and a verdict: pass/fail/equivocal. By Ch 18 the reader has produced a report that *no AI evaluation lab is currently producing* — a multi-axis cognitive assessment grounded in comparative cognition rather than NLP benchmarks.

**Why it fits this book:** Ch 17 is exactly this exercise applied once. Project Option 4 is Ch 17 applied 18 more times. The book is building toward a diagnostic stance on AI; this project externalizes that stance into a working report. It also produces something publishable — a "what current AI actually does on cognition's natural axes" report has real audience demand.

**Adaptability:** The reader picks which AI system to test. A Claude user will produce one report; a multi-model evaluator will produce a comparison. The chapter capacities are fixed; the test subject is the variable.

**Tool path:** Mix. Claude chat for running the diagnostics in conversation. Claude Code at Ch 17 if the reader wants to script Ullman-style perturbation batteries. A Claude Project to maintain context across the 19 entries (the reader will want the system to remember their evaluation criteria and prior verdicts).
